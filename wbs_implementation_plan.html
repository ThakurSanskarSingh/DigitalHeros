UI/UX Live charity impact counter on the homepage

Instead of a static number, show a real-time animated counter: "£24,318 donated to 7 charities this year" — incrementing live using Supabase Realtime. This is the emotional gut-punch that makes someone subscribe immediately. No other golf platform does this.

Draw UX Theatrical draw reveal screen

When a draw is published, don't just show a results table. Build a number-reveal animation — numbers flip one by one like a slot machine, with match highlighting and a confetti burst for jackpot winners. This makes the monthly draw feel like an event, not a database query.

Score UX Score trend sparkline + personality tags

Show a tiny sparkline next to the score list showing their form trend. Add auto-generated tags like "Hot streak" or "On the up" based on score direction — purely frontend logic. Makes the score entry screen feel alive and personal, not like filling a form.

Tech Server Actions + optimistic UI for score entry

Use Next.js 14 Server Actions with useOptimistic — when a user adds a score, it appears instantly in the list while the server processes it in the background. Most interns won't implement this. It shows deep Next.js knowledge and results in a snappy, native-feeling UX.

Admin Draw simulation with frequency heatmap

In admin's simulation mode, visualise all user scores as a heatmap (1–45) showing which numbers appear most. When switching to the weighted algorithm, show in real time how the draw distribution shifts. This turns an abstract feature into something an evaluator can interact with and be impressed by.

Code quality Write unit tests for the draw engine

Most interns skip this. Write Vitest tests for: the rolling 5-score logic, the match-detection function (3/4/5), prize split with multiple winners, and jackpot rollover edge case. Include a README section showing test output. Evaluators who look at code will immediately notice this.

Architecture Scores snapshot in draw_entries

When a draw runs, snapshot each user's 5 scores at that moment into the draw_entries.scores_snapshot JSONB column. This means draw results are always historically accurate even if a user later edits their scores. This is a subtle data-modelling decision most people miss — and it directly shows "scalability thinking" from the eval criteria.
